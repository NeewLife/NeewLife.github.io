[Github] README 파일에서 새 창 열기가 될까?

2023/04/20 카테고리 Git

배경

블로그를 작업하려 이것저것 끄적이다가 블로그 주소를 README 파일에 올려서 이동할 수 있게 하고 싶었다.

방법

GitHub는 Markdown 문법을 사용하니 HTML의 a 태그를 사용해봤다.

  • html을 작성할 때는 a 태그에 target="_blank" 를 사용하고,
  • md파일을 작성할 때는 {:target=”_blank”} 를 사용한다.

여기서는 예시로 Google 창이 열리게 해보았다.

그런데 문제가 하나 있었다….
GitHub 페이지 엔진이 Kramdown 이라는 또 다른 Markdown 방식을 사용한다는 것이다.

그래서 GitHub README 파일에서는 {:target=”_blank”} 이 코드가 먹히지 않는다..

StackOverflow 관련 답변

여담

구글링해서 찾아보니까 예전부터 “이걸 새 창으로 띄우는 방식으로 만들어야 돼? 말아야 돼?” 하는 토론이 있었던 것 같다.

https://github.com/okjsp/okky/issues/63

입장이 두 개로 나뉘는데,

  • 사용자한테 왜 CTRL + 클릭 방식으로 선택지를 안 주고 너가 강제로 새 창을 띄우려 하느냐 라는 입장
    (관련링크 : Top 10 Mistakes in Web Design 에서 9번째)
  • CTRL + 클릭 방식을 일반적인 사용자는 모른다. 사용자의 편의를 위해서 우리같은 개발자들이 추가 해주는 게 낫다는 입장
    (관련링크 : Why external links should open in new-tabs)

💡 전자의 입장은
  • 뒤로가기 버튼이 비활성화된 채로 사이트에 머물게 하는 것은 굉장히 자멸적인 전략이다
  • 사용자는 종종 새로운 창이 열렸다는 것을 인식하지 못한다, 특히 모니터가 작을 경우 더욱 그렇다.
  • 그러므로 계속 여러개의 창이 열리게 되면 오염되는 기분을 느낄 것이다.
💡 후자의 입장은
  • 브라우저 탭의 도입으로 예전보다 새 창을 새 탭으로 대신하여 관리하기가 쉬워졌다.
  • 자기가 뒤로 다시가고 싶은 사이트로 뒤로가기 버튼을 계속 누르게 되면 피로도가 쌓일 것이다.
  • 또한, 웹사이트의 리소스를 사용하기 때문에 자원적으로도 비효율적이다.

전자의 입장과 후자의 입장이 팽팽한 것 같다. 꽤 오랜 시간이 지났는데도 불구하고 결론이 나지 않은 것 같다.
이게 모바일이나 웹에 친숙한 10~30대 정도면 몰라도 50~60대 분들도 인터넷을 사용하니까…

실제로 전자의 입장측에서 예시를 들었다.
자기 아버지가 컴퓨터를 잘 못하는데, 판매사이트가 물품 검색중에 새 탭을 열어주었나보다.
그런데 아버지는 다시 있던 사이트로 돌아가려 했는데, 뒤로가기 버튼이 비활성화된 것을 보고 혼란스러워하다가 그냥 컴퓨터를 끄고 매장에 직접 방문하기로 했다고 한다.
그러면 그 판매사이트는 구매력을 잃게 된 것이지 않느냐 하면서 반박하는 내용도 있더라.

느낀 점

웹 페이지의 제작과정에서 사용자의 UX를 편리하게 하기 위해 이렇게 찾아봤는데,
생각보다 많은 사람들이 UX를 편리하게 하기 위해서 토론하는 모습들을 보면서 재밌었다. (역시 싸움구경이 꿀잼)

UI/UX는 언제나 고민해도 어려운 것 같다. 계속 공부해서 익숙해지는 길을 가려고 노력해야지
